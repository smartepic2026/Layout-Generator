# 프로젝트 판단 기준 (이 파일을 매 결정 전에 적용할 것)

## 최우선 원칙 — 무엇을 기준으로 선택할 것인가

이 프로젝트의 모든 기술 선택·추천은 아래 순서의 기준으로 판단한다.
**"빠르다 / 코드가 적다 / 구현이 쉽다"는 추천 사유가 될 수 없다.**

1. **최종 결과물의 성능** — "돌아간다"가 아니라 "실제 도면을 넣었을 때
   GMP 전문가가 납득할 배치가 나오는가". 휴리스틱보다 수학적으로 방어
   가능한 방법을 택한다.
2. **실무에서 쓸 수 있는 의미 있는 수준** — 데모용/장난감이 아니라
   현업이 실제로 쓸 수 있는 견고함(더러운 실데이터 처리, 재현성, 검증).
3. **SW 등록 · 특허** — 신규하고 비자명한 기술 구성을 만든다. 무엇이
   청구항이 될 수 있는지 항상 의식한다.
4. **연구 논문** — 재현가능한 파이프라인, 정량 평가지표(violation 수,
   점수 개선율, 전문가 배치와의 일치도), 근거(ISPE/ISO/EU GMP) 추적성.

시간·노력이 더 들더라도 위 기준에서 더 나은 쪽을 택한다.
일정 압박이 있으면 "빠른 안으로 바꾸자"가 아니라, 사용자에게
트레이드오프를 보고하고 결정을 받는다.

## 위 기준에서 이미 내려진 결정 (번복하려면 사용자 승인 필요)

- **솔버는 CP-SAT(OR-tools).** SA·그리디 아님. 이유: 결정론적(재현성),
  수학적 최적해(논문·특허 방어), 규칙→정수제약 컴파일이 특허 핵심.
- **엑셀은 직접 파싱.** 사람이 CSV로 손변환하지 않는다. 원본 .xlsx를
  파싱 + 더러운 데이터(오타행/중량누락) 정제 + source 태그. 재현가능해야 함.
- **데이터·수식·베이스라인이 솔버보다 먼저.** 순서: (a) sort_order/
  bbox_m/connects_to 채움 → (b) P1·P2·P6·P7 수식 + 현재 strip-band
  점수를 before 베이스라인으로 측정·기록 → (c) CP-SAT 솔버 +
  constraint_compiler → (d) infeasible fallback.
- **P3·P5·P8 점수는 1차 보류(삭제 아님, 중립).** 검수자가 없어 추론
  데이터(open_closed 등)로 점수를 오염시키지 않는다. 스키마 필드는 미리
  두고 값만 null. 2차에서 값 채우고 가중치만 켜면 살아나게. 이 "근거가
  명확한 제약만 적용하고 불확실한 건 confidence 태깅해 분리"는
  버그가 아니라 의도된 설계 원칙(epistemic honesty)이며 논문/특허 강점.

## 코드 경계 (절대 규칙)

- `src/rule_engine/`는 팀원 소유. **읽기만, 수정 금지.**
- 내 작업 범위는 `src/drawing_agent/` + scoring + 동선검증.
- 두 영역은 공유 JSON 스키마(계약)로만 통신. 서로 import 금지.

## 설계 결정 로그 (논문 method / 특허 청구항 원재료)

중요한 설계 결정을 내릴 때마다 "무엇을·왜·어떤 근거(ISPE/ISO 조항)로"
를 rationale 로그에 남긴다. 나중에 논문 method 섹션과 특허 명세서의
원재료가 된다.

## 작업 기록 규칙 (매 세션 자동 — 시키지 않아도 수행)

세션이 바뀌면(나갔다 들어오면) 이전 대화 맥락은 사라진다. 따라서
진행 상황을 **반드시 파일로** 남겨, 다음 세션이 읽고 이어갈 수 있게 한다.

### 사용하는 기록 파일 (새로 만들지 말고 아래 파일을 쓴다)
- `docs/PROGRESS.md` — 진행 상황 (어디까지 했나)
- `docs/decisions.md` — 설계 결정 로그 (왜 + 근거) → 논문 method·특허 원재료
- `prompts.md` — **(루트에 이미 존재)** 사용자가 준 중요한 지시·방향 전환을
  시간순으로 누적. 새 파일(prompt_log.md 등) 만들지 말고 이 파일에 이어 쓴다.
- `README.md` — 모듈/실행법/산출물 구조 + **커밋 한 줄 로그**(아래)

### 세션 시작 시 (매번 가장 먼저)
1. `docs/PROGRESS.md` 를 읽어 지금까지 어디까지 했는지 파악한다.
2. `docs/decisions.md` 를 읽어 이미 내려진 결정을 확인한다(번복 금지).
3. 그 다음 작업을 이어간다. 처음부터 다시 시작하지 않는다.

### 작업 진행 중 — 발생 즉시 기록
- **PROGRESS.md**: Phase/작업 하나 끝낼 때마다 한 줄.
  `[날짜] PhaseX.Y 완료 — 무엇을 했고 다음 할 일은 무엇`. 중간에 멈출 때도
  "현재 여기까지, 다음은 이것" 을 남긴다.
- **decisions.md**: 설계 결정마다 `결정 / 이유 / 근거(ISPE·ISO 조항) / 날짜`.
- **prompts.md**: 사용자의 중요한 지시·방향 전환을 시간순으로 누적
  (어떤 요구로 무엇이 바뀌었는지 추적 — 감사·논문 재현용).

### 커밋 규칙 (기존 관습 유지)
- **커밋할 때마다 `README.md` 에 변경 요약을 한 줄 추가한다.**
  형식: `- [날짜/커밋] 무엇을 바꿨는지 한 줄`. (기존에 README에 커밋
  로그를 한 줄씩 남기던 방식을 그대로 이어간다.)
- 모듈/실행법/산출물 구조가 바뀌면 README 본문도 현재 코드 상태와
  일치하도록 갱신한다.

### 기록 우선순위
- 코드를 짜기 전에 PROGRESS/decisions 를 읽고, 짠 후에 갱신한다.
- "기록은 나중에" 금지. 작업과 동시에 남긴다. 세션이 끊겨도 파일만
  보면 누구든(다음 세션의 나 포함) 이어갈 수 있어야 한다.
